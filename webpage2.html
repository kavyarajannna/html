System Requirements Analysis
Process ID SYS.2
Process name System Requirements Analysis
Process purpose The purpose of the System Requirements Analysis Process is to transform the defined stakeholder requirements into a set of system requirements that will guide the design of the system.
Process outcomes As a result of successful implementation of this process:
1) A defined set of system requirements is established;
2) System requirements are categorized and analyzed for correctness and verifiability;
3) The impact of system requirements on the operating environment is analyzed;
4) Prioritization for implementing the system requirements is defined;
5) The system requirements are updated as needed;
6) Consistency and bidirectional traceability are established between stakeholder requirements and system requirements;
7) The system requirements are evaluated for cost, schedule, and technical impact;
8) The system requirements are agreed and communicated to all affected parties.
Base Practices SYS.2.BP1: Specify system requirements. Use stakeholder requirements to identify system functions and capabilities. Specify functional and non-functional system requirements in a system requirements specification. [OUTCOME 1, 5, 7]

NOTE 1: Application parameters influencing functions and capabilities are part of the system requirements.
NOTE 2: For changes to stakeholder requirements, SUP.10 applies.

SYS.2.BP2: Structure system requirements. Structure requirements in the system requirements specification by grouping, sorting, categorizing, and prioritizing according to stakeholder needs. [OUTCOME 2, 4]

NOTE 3: Prioritizing typically includes assigning functional content to planned releases. Refer to SPL.2.BP1.

SYS.2.BP3: Analyze system requirements. Analyze specified requirements including interdependencies to ensure correctness, feasibility, verifiability, and support risk identification. Analyze impact on cost, schedule, and technical feasibility. [OUTCOME 1, 2, 7]

NOTE 4: Analysis of impact on cost and schedule supports adjustment of project estimates. Refer to MAN.3.BP5.

SYS.2.BP4: Analyze impact on operating environment. Identify system interfaces and analyze impact of system requirements on these interfaces and the environment. [OUTCOME 3, 7]

SYS.2.BP5: Develop verification criteria. Define qualitative and quantitative measures for verifying each system requirement. [OUTCOME 2, 7]

NOTE 5: Verification criteria demonstrate requirement verification within agreed constraints and inform system test case development. Verification not covered by testing is addressed in SUP.2.

SYS.2.BP6: Establish bidirectional traceability. Ensure traceability between stakeholder requirements and system requirements. [OUTCOME 6]

NOTE 6: Traceability supports coverage, consistency, and impact analysis.

SYS.2.BP7: Ensure consistency. Maintain consistency between stakeholder requirements and system requirements. [OUTCOME 6]

NOTE 7: Consistency is supported by traceability and can be demonstrated by review records.

SYS.2.BP8: Communicate agreed system requirements. Communicate system requirements and updates to relevant parties. [OUTCOME 8]
